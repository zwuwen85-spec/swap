# 校园闲置物品交换平台 - 数据库设计

## 一、数据库概述

- 数据库类型：MySQL 8.0+
- 字符集：utf8mb4
- 排序规则：utf8mb4_unicode_ci
- 命名规范：小写+下划线

---

## 二、数据表设计

### 1. 用户表 (user)

用户基础信息和认证信息。

| 字段名 | 类型 | 长度 | 允许NULL | 默认值 | 说明 |
|--------|------|------|----------|--------|------|
| id | BIGINT | - | NO | - | 用户ID（主键，自增） |
| username | VARCHAR | 50 | NO | - | 用户名（唯一） |
| password | VARCHAR | 100 | NO | - | 密码（bcrypt加密） |
| email | VARCHAR | 100 | YES | NULL | 邮箱（唯一） |
| phone | VARCHAR | 20 | YES | NULL | 手机号（唯一） |
| avatar | VARCHAR | 255 | YES | NULL | 头像URL |
| nickname | VARCHAR | 50 | YES | NULL | 昵称 |
| gender | TINYINT | - | YES | NULL | 性别（0未知 1男 2女） |
| student_id | VARCHAR | 50 | YES | NULL | 学号 |
| school | VARCHAR | 100 | YES | NULL | 学校名称 |
| status | TINYINT | - | NO | 1 | 状态（0禁用 1正常 2冻结） |
| credit_score | INT | - | NO | 100 | 信誉分（0-100） |
| create_time | BIGINT | - | NO | - | 创建时间（时间戳） |
| update_time | BIGINT | - | YES | NULL | 更新时间（时间戳） |
| last_login_time | BIGINT | - | YES | NULL | 最后登录时间 |
| last_login_ip | VARCHAR | 50 | YES | NULL | 最后登录IP |

**索引：**
- PRIMARY KEY (id)
- UNIQUE KEY uk_username (username)

- INDEX idx_status (status)

## 三、ER图说明

### 主要关系

```
用户 (user)
    │
    ├─ 1:N → 商品 (goods)        # 一个用户可以发布多个商品
    ├─ 1:N → 收藏 (favorite)     # 一个用户可以收藏多个商品
    ├─ 1:N → 评论 (comment)      # 一个用户可以发表多个评论
    ├─ 1:N → 交换发起 (exchange) # 作为发起人
    ├─ 1:N → 交换接收 (exchange) # 作为接收人
    └─ 1:N → 消息 (message)      # 发送/接收消息

商品 (goods)
    │
    ├─ N:1 → 分类 (category)     # 一个商品属于一个分类
    ├─ 1:N → 收藏 (favorite)     # 一个商品可以被多个用户收藏
    ├─ 1:N → 评论 (comment)      # 一个商品可以有多条评论
    └─ 1:N → 交换 (exchange)     # 一个商品可以有多条交换记录

分类 (category)
    │
    └─ 1:N → 子分类             # 自关联，支持多级分类
```

## 五、数据字典说明

### 状态码规范

- **商品状态 (goods.status)**：
  - 0: 已下架
  - 1: 在售
  - 2: 已售出
  - 3: 已交换

- **商品类型 (goods.type)**：
  - 1: 售卖
  - 2: 交换
  - 3: 均可

- **交换状态 (exchange.status)**：
  - 0: 待处理
  - 1: 已接受
  - 2: 已拒绝
  - 3: 已取消
  - 4: 已完成

- **用户状态 (user.status)**：
  - 0: 禁用
  - 1: 正常
  - 2: 冻结

### 价格单位

所有价格字段（price、original_price）统一使用**分**作为单位，避免浮点数计算问题。

前端展示时需要除以100转换为元。

## 六、性能优化建议

### 1. 索引策略

- 为高频查询字段建立索引
- 组合索引遵循最左前缀原则
- 避免过多索引影响写入性能

### 2. 分表分库

当数据量达到百万级时考虑：
- 按时间分表（message、operation_log）
- 按用户ID哈希分表（goods、comment）

### 3. 读写分离

- 主库负责写操作
- 从库负责读操作
- 使用中间件实现自动路由

---

## 七、数据安全

### 1. 敏感字段加密

- password: bcrypt（cost=10）
- phone: AES加密（可选）

### 2. 软删除

重要表添加 `deleted_at` 字段，实现软删除：
- user、goods、comment 表

### 3. 数据备份

- 每日全量备份
- 实时增量备份
- 保留30天历史数据
