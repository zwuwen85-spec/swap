# 聊天模块测试指南

## 测试环境准备

1. 确保后端服务已启动（支持WebSocket）
2. 确保Redis服务已启动（在线状态管理）
3. 准备至少两个测试账号用于互相发送消息

## 1. WebSocket连接测试

### 1.1 连接建立

**测试步骤：**
1. 用户A登录系统
2. 打开浏览器开发者工具 → Network → WS
3. 访问消息列表页面
4. 观察是否建立了WebSocket连接

**预期结果：**
- WebSocket连接成功建立（状态码101）
- 连接URL格式：`ws://localhost:8080/api/v1/ws/chat`
- 连接携带JWT token进行认证

### 1.2 心跳测试

**测试步骤：**
1. 保持WebSocket连接打开
2. 观察Network面板中的WebSocket帧

**预期结果：**
- 每54秒发送一次ping消息
- 收到pong响应
- 连接保持活跃状态

### 1.3 在线状态测试

**测试步骤：**
1. 用户A登录，进入聊天页面
2. 用户B登录，查看与用户A的聊天
3. 用户A断开连接/关闭页面

**预期结果：**
- 用户A在线时，用户B看到"在线"标签
- 用户A离线时，用户B看到"离线"标签
- Redis中存在`online:user:{userID}`键，TTL为300秒

## 2. 消息发送测试

### 2.1 发送文本消息

**测试步骤：**
1. 用户A和用户B互相登录
2. 用户A向用户B发送消息："你好，请问这个商品还在吗？"
3. 观察双方界面

**预期结果：**
- 用户A：消息显示在右侧（蓝色气泡）
- 用户B：实时收到消息，显示在左侧（白色气泡）
- 消息包含：发送者ID、接收者ID、内容、类型、时间戳
- 数据库messages表中新增一条记录

**验证数据库：**
```sql
SELECT * FROM messages WHERE sender_id = ? AND receiver_id = ? ORDER BY create_time DESC LIMIT 1;
```

### 2.2 发送空消息

**测试步骤：**
1. 在输入框中只输入空格
2. 点击发送按钮

**预期结果：**
- 发送按钮处于禁用状态（disabled）
- 或显示错误提示："消息内容不能为空"

### 2.3 发送给不存在用户

**测试步骤：**
1. 直接访问聊天页面：`/chat/99999`
2. 尝试发送消息

**预期结果：**
- 显示错误提示："接收用户不存在"
- 或接口返回404错误

### 2.4 发送消息给自己

**测试步骤：**
1. 修改请求参数，将receiver_id设为当前用户ID
2. 发送消息

**预期结果：**
- 接口返回错误："不能发送消息给自己"
- 数据库中不会保存消息

## 3. 消息接收测试

### 3.1 实时消息接收

**测试步骤：**
1. 用户A和用户B同时在线，处于聊天页面
2. 用户A发送消息
3. 计时

**预期结果：**
- 用户B在100ms内收到消息
- 消息自动滚动到底部
- 收到消息时显示"刚刚"时间

### 3.2 离线消息

**测试步骤：**
1. 用户B离线
2. 用户A发送多条消息
3. 用户B上线，进入聊天

**预期结果：**
- 用户B能看到所有离线期间的消息
- 消息按时间正序排列
- 未读消息数为发送的消息数量

### 3.3 消息已读状态

**测试步骤：**
1. 用户A发送消息
2. 用户B打开聊天页面
3. 查询数据库

**预期结果：**
- 消息的is_read字段更新为1
- 或在用户B实时查看时标记为已读

## 4. 会话列表测试

### 4.1 会话列表显示

**测试步骤：**
1. 用户A与多个用户发送消息
2. 访问消息列表页面

**预期结果：**
- 显示所有有过对话的用户
- 每个会话显示：头像、昵称、最后一条消息、时间、未读数
- 按最后消息时间倒序排列

### 4.2 未读消息计数

**测试步骤：**
1. 用户A向用户B发送3条消息
2. 用户B不查看，直接查看消息列表

**预期结果：**
- 用户与用户A的会话显示红色徽章"3"
- 顶部显示总未读数

**验证API：**
```bash
curl -X GET "http://localhost:8080/api/v1/message/unread-count" -H "Authorization: Bearer {token}"
```

### 4.3 点击进入聊天

**测试步骤：**
1. 在消息列表点击某个会话

**预期结果：**
- 跳转到聊天页面：`/chat/{user_id}`
- 加载历史消息
- 清除该会话的未读数

## 5. 消息列表测试

### 5.1 加载历史消息

**测试步骤：**
1. 进入与某用户的聊天页面
2. 观察消息列表

**预期结果：**
- 显示所有历史消息
- 最新消息在最底部
- 自动滚动到最新消息

### 5.2 分页加载（如实现）

**测试步骤：**
1. 消息数量超过单页限制（如50条）
2. 滚动到顶部
3. 继续向上滚动

**预期结果：**
- 触发加载更早的消息
- 新消息追加到列表顶部
- 滚动位置保持

### 5.3 时间显示格式

**测试步骤：**
1. 观察不同时间的消息

**预期结果：**
- 1分钟内：显示"刚刚"
- 1小时内：显示"X分钟前"
- 24小时内：显示"X小时前"
- 更早：显示"MM-DD HH:mm"

## 6. 异常情况测试

### 6.1 网络断开

**测试步骤：**
1. 打开聊天页面
2. 断开网络连接
3. 尝试发送消息

**预期结果：**
- 显示网络错误提示
- WebSocket连接关闭
- 恢复网络后自动重连

### 6.2 并发发送

**测试步骤：**
1. 用户A快速连续发送10条消息
2. 观察用户B接收情况

**预期结果：**
- 所有消息按顺序保存到数据库
- 用户B按顺序接收所有消息
- 无消息丢失或重复

### 6.3 超长消息

**测试步骤：**
1. 输入超长文本（如10000字符）
2. 发送消息

**预期结果：**
- 或拒绝发送（前端验证）
- 或成功发送但界面正常显示
- 不导致内存溢出或崩溃

### 6.4 特殊字符

**测试步骤：**
1. 发送包含特殊字符的消息：emoji、HTML标签、SQL注入语句
2. 观察显示

**预期结果：**
- emoji正常显示
- HTML标签被转义（不执行）
- SQL语句作为纯文本显示
- 无XSS或SQL注入风险

## 7. 性能测试

### 7.1 并发连接

**测试步骤：**
1. 使用压力测试工具（如JMeter）
2. 模拟1000个并发WebSocket连接

**预期结果：**
- 服务器稳定运行
- 内存使用在合理范围
- 响应时间<100ms

### 7.2 消息吞吐量

**测试步骤：**
1. 模拟每秒发送100条消息
2. 持续5分钟

**预期结果：**
- 无消息丢失
- 数据库写入正常
- WebSocket推送延迟<50ms

### 7.3 数据库查询优化

**测试步骤：**
1. 查看慢查询日志
2. 测试消息列表查询性能

**预期结果：**
- 所有查询<100ms
- 使用了合适的索引：
  - idx_sender_receiver_time (sender_id, receiver_id, create_time)
  - idx_receiver_read (receiver_id, is_read)

## 常见问题排查

### 问题1：WebSocket连接失败

**可能原因：**
- JWT token缺失或过期
- 后端服务未启动
- 防火墙阻止WebSocket连接

**解决方法：**
- 检查Network面板，查看握手请求
- 确认token有效性
- 检查后端日志

### 问题2：消息不实时

**可能原因：**
- WebSocket断开
- 消息推送失败
- 前端未处理事件

**解决方法：**
- 检查WebSocket连接状态
- 查看后端Hub的Broadcast日志
- 检查前端消息监听器

### 问题3：未读数不准确

**可能原因：**
- 未读状态未及时更新
- 缓存未刷新
- 多设备同步问题

**解决方法：**
- 检查MarkAsRead逻辑
- 查询数据库验证is_read字段
- 实现多设备实时同步
