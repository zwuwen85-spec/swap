# 通知模块测试指南

## 测试环境准备

1. 确保后端服务已启动
2. 确保数据库服务正常
3. 准备测试账号和关联数据（商品、交换、评论）

## 1. 通知创建测试

### 1.1 系统通知创建

**测试步骤：**
1. 管理员创建系统通知
2. 接收用户ID：123
3. 类型：1（系统通知）
4. 标题："系统维护通知"
5. 内容："系统将于今晚22:00进行维护"

**预期结果：**
- 通知成功创建
- 数据库notification表新增记录
- user_id为123，type为1

**验证数据库：**
```sql
SELECT * FROM notification WHERE user_id = 123 AND type = 1;
```

### 1.2 交换通知自动创建

**测试步骤：**
1. 用户A发起与用户B的商品交换
2. 检查用户B是否收到通知

**预期结果：**
- 用户B收到"新的交换请求"通知
- type为2（交换通知）
- title为"新的交换请求"
- content包含商品标题
- link指向交换详情

**触发场景：**
- 发起交换请求（卖家收到）
- 交换被接受（买家收到）
- 交换被拒绝（买家收到）
- 交换已完成（双方收到）

### 1.3 评论通知自动创建

**测试步骤：**
1. 用户A评论用户B的商品
2. 检查用户B是否收到通知

**预期结果：**
- 用户B收到"新评论通知"
- type为3（评论通知）
- title为"新评论通知"
- content包含评论者名称和商品标题

## 4. 标记已读测试

### 4.1 标记单条为已读

**测试步骤：**
1. 调用API：PUT /api/v1/notification/read/123

**预期结果：**
- 通知的is_read变为1
- 未读数量-1

**验证数据库：**
```sql
SELECT is_read FROM notification WHERE id = 123;

```

**验证数据库：**

### 4.4 标记已读后再点击

**测试步骤：**
1. 通知已经是已读状态

**预期结果：**

## 6. 通知触发场景测试

### 6.1 交换请求场景

**完整流程：**
1. 用户A发起交换 → 用户B收到"新的交换请求"
2. 用户B接受 → 用户A收到"交换请求已接受"
3. 用户B拒绝 → 用户A收到"交换请求已拒绝"
4. 双方完成 → 双方收到"交换已完成"

**验证点：**
- 每个动作都触发通知
- 接收者正确
- 通知内容准确
- link可跳转

### 6.2 评论场景

**完整流程：**
1. 用户A评论用户B的商品 → 用户B收到"新评论通知"
2. 用户C回复用户A的评论 → 用户A收到回复通知（如实现）

**验证点：**
- 只有主评论触发通知
- 卖家收到通知
- 评论者名称正确

### 6.3 系统通知场景

**触发场景：**
- 系统维护
- 活动公告
- 账户异常
- 功能更新

**验证点：**
- type=1
- 所有关联用户都能收到

## 常见问题排查

### 问题1：通知未触发

**可能原因：**
- 通知逻辑未调用
- 异步操作失败
- 数据库连接问题

**解决方法：**
```sql
-- 检查通知是否创建
SELECT * FROM notification WHERE user_id = ? ORDER BY create_time DESC LIMIT 10;
-- 检查应用日志
```

### 问题2：未读数量不准确

**可能原因：**
- 并发标记已读
- 缓存未更新
- 统计逻辑错误

**解决方法：**
```sql
-- 手动统计未读数
SELECT COUNT(*) FROM notification WHERE user_id = ? AND is_read = 0;
-- 与API返回值对比
```

### 问题3：通知跳转失败

**可能原因：**
- link为空
- link格式错误
- 路由不存在

**解决方法：**
- 确保link格式正确：/exchange/123
- 验证路由是否存在
- link为空时不跳转
